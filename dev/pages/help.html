Introduction

Juxtaphor is a web interface that allows users to 1) save different versions of a text 2) create collections from those versions and 3) generate dynamic visualizations from collections.

This project is a collaboration between Sibelan Forrester, Professor of Russian Literature, and the SPEED program at Swarthmore College. It was built by students Alexander Simms, Benjamin Xie, Katherine Hamilton, and Andrew Pak. All code released under the BSD license, available on our Github VisualizingTranslations repository.

Sources

The first step to using this site is to add source files to your workspace. Source files serve as the raw data that the user can enter. After entering the sources, one can view what has been uploaded by clicking on the orange button with an eye icon under the sources tab. They can also be deleted by click on the trashcan button. The whole point of uploading sources, however, is to transform them into smarter texts that can be visualized. Checking off sources and hitting transform will convert the sources into witnesses.

Witnesses

After transforming the sources, we have successfully used Juxta to prepare the witnesses. Witnesses are versions of the original text that we want to eventually compare. When a source is transformed, it gets extra metadata that allows Juxta to eventually collate and visualize the different versions. Under the witnesses tab, check off the witnesses you want to compare and collate them. Collating groups the witnesses together into collection sets, which are easily visualized.

Collections

Collections are sets of witnesses that can be visualized. Under the collections tab, one can view all the collections that have been built. Some may take a second to finsih collating. Once collated, however, the collection can be visualized as either a heatmap or a side-by-side comparison view.

TEI Parallel Segmentation

The Juxta API relies on TEI to parse witnesses and serve visualizations based on collated sets. In particular, Juxta uses TEIs Parallel Segmentation to encode text variations. Roughly speaking, Parallel Segmentation allows users to align portions of several different texts with each other. It does not hardcode one single "base text" against which all others variants are compared. Instead it allows for multiple readings and fluid recombination between texts.

The Juxta API automatically performs alignment and markup using Parallel Segmentation; however, if you wish to have more fine-grained control or already have a collated set, you can import an existing XML that encodes variant texts in TEI using Parallel Segmentation. To do so, cut and paste valid TEI XML into the text input area, click the XML and TEI Parallel Segmentation checkboxes that appear below the text input area, and save source as you normally would. This will automatically generate separate witnesses, add a new collection from those witnesses, and collate that collection. This may take a few moments to complete.

Please note that in your TEI XML, please ensure that the titles you assign to your witnesses in your witness list within your tags must be unique. If any witnesses already exist in your workspace with the same titles, your new set will not be able to collate. If this happens, simply delete the set, any witnesses that were generated, and the source -- then try to upload again.
